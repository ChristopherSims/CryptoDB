# CryptoDB: Cryptographic Database with Immutable Audit Ledger

**Goal:** Build a secure, queryable database where all stored data is cryptographically protected at rest and in transit, and every access (read, write, delete, key rotation) is recorded in a tamper-evident audit ledger.

**Location:** `C:\Users\sims\Desktop\CryptoDB`

---

## 1. Current Context / Assumptions

- Greenfield project вҖ” no existing code or schema.
- Target stack is flexible; this plan recommends Python (backend) + SQLite/PostgreSQL (metadata) + file-based encrypted blobs (data) for simplicity and portability.
- Single-node deployment initially; distributed consensus deferred to Phase 3.
- User wants features *beyond* basic crypto + ledger вҖ” this plan includes RBAC, secure querying, anomaly detection, and compliance hooks.

## 3. Step-by-Step Implementation Plan

### Phase 0: Foundation & Tooling (1вҖ“2 days)

| # | Task | Files |
|---|------|-------|
| 0.1 | Initialize Python project (`pyproject.toml`, `ruff`, `pytest`, `mypy`) | `pyproject.toml`, `.python-version` |

| 0.5 | Add configuration management (Pydantic Settings, env vars) | `src/cryptodb/config.py` |

### Phase 1: Cryptographic Core (3вҖ“4 days)

| # | Task | Details | Files |
|---|------|---------|-------|
| 1.1 | **Envelope Encryption** | Data keys (DEK) encrypt payloads; DEKs encrypted by master key (KEK). | `src/cryptodb/crypto/envelope.py` |
| 1.2 | **Master Key Management** | KEK stored in OS keyring or environment; support for AWS KMS / HashiCorp Vault interfaces. | `src/cryptodb/crypto/keystore.py` |
| 1.3 | **Algorithm Selection** | AES-256-GCM for data, XChaCha20-Poly1305 as alt, PBKDF2/Argon2 for key derivation. | `src/cryptodb/crypto/ciphers.py` |
| 1.4 | **Key Rotation** | Automatic DEK rotation on write; re-encryption of old records on access or background job. | `src/cryptodb/crypto/rotation.py` |
| 1.5 | **Crypto-Shredding** | Secure deletion = shred DEK; data becomes irrecoverable without wiping storage blocks. | `src/cryptodb/crypto/shredding.py` |
| 1.6 | **Integrity** | HMAC-SHA256 or AEAD tag on every record; verify before decrypt. | `src/cryptodb/crypto/integrity.py` |

### Phase 2: Data Storage Layer (2вҖ“3 days)

| # | Task | Details | Files |
|---|------|---------|-------|
| 2.1 | **Metadata Schema** | Tables: `records`, `data_keys`, `users`, `policies`. Store only ciphertext + non-sensitive metadata. | `src/cryptodb/db/metadata.py` |
| 2.2 | **Encrypted Blob Store** | Store ciphertext blobs on filesystem or S3-compatible backend; metadata DB holds pointers. | `src/cryptodb/storage/blob.py` |
| 2.3 | **Secure Querying** | Deterministic AES-SIV or HMAC-based indexing for equality queries on encrypted fields. | `src/cryptodb/crypto/searchable.py` |
| 2.4 | **Record Versioning** | Keep N versions of each record; rollback capability. | `src/cryptodb/db/versions.py` |
| 2.5 | **Compression** | Compress before encrypt (zlib/zstd) to reduce storage footprint. | `src/cryptodb/storage/compression.py` |

### Phase 3: Immutable Audit Ledger (3вҖ“4 days)

| # | Task | Details | Files |
|---|------|---------|-------|
| 3.1 | **Ledger Schema** | Table: `audit_log` with entry_id, timestamp, actor, action, record_id, result, hash. | `src/cryptodb/ledger/schema.py` |
| 3.2 | **Merkle Tree / Hash Chain** | Each entry includes `previous_hash`; merkle root periodically signed by a private key. | `src/cryptodb/ledger/chain.py` |
| 3.3 | **Tamper Detection** | Background verification task; alert if hash chain breaks. | `src/cryptodb/ledger/verify.py` |
| 3.4 | **Ledger Export** | Export to WORM storage, JSONL, or blockchain anchoring for external notarization. | `src/cryptodb/ledger/export.py` |
| 3.5 | **Anomaly Detection** | Flag unusual access patterns (off-hours, bulk downloads, unknown IPs). | `src/cryptodb/ledger/anomaly.py` |

### Phase 4: Access Control & Authentication (2вҖ“3 days)

| # | Task | Details | Files |
|---|------|---------|-------|
| 4.1 | **User Management** | Local users or LDAP/OAuth2 integration; password hashing with Argon2. | `src/cryptodb/auth/users.py` |
| 4.2 | **RBAC** | Roles: `admin`, `writer`, `reader`, `auditor`. | `src/cryptodb/auth/rbac.py` |
| 4.3 | **ABAC Extensions** | Attribute-based rules: time-of-day, IP range, data classification level. | `src/cryptodb/auth/abac.py` |
| 4.4 | **ACL per Record** | Each record can have an explicit ACL overriding role defaults. | `src/cryptodb/auth/acl.py` |
| 4.5 | **Session Management** | Short-lived JWTs, refresh tokens, mandatory MFA for admin actions. | `src/cryptodb/auth/session.py` |

### Phase 5: API & SDK (2вҖ“3 days)

| # | Task | Details | Files |
|---|------|---------|-------|
| 5.1 | **REST API** | FastAPI-based: `POST /records`, `GET /records/{id}`, `DELETE /records/{id}`, `POST /query`, `GET /audit`. | `src/cryptodb/api/main.py` |
| 5.2 | **OpenAPI + Docs** | Auto-generated Swagger UI, but hide internal endpoints. | `src/cryptodb/api/routes.py` |
| 5.3 | **Python SDK** | Typed client library for local scripting. | `src/cryptodb/sdk/client.py` |
| 5.4 | **CLI Tool** | `cryptodb put`, `cryptodb get`, `cryptodb audit`, `cryptodb rotate`. | `src/cryptodb/cli/main.py` |

### Phase 6: Advanced Features (Stretch вҖ” 4вҖ“5 days)

| # | Feature | Description | Files |
|---|---------|-------------|-------|
| 6.1 | **Proxy Re-encryption** | Allow secure data sharing without revealing DEK to proxy. | `src/cryptodb/crypto/pre.py` |
| 6.2 | **Homomorphic Encryption (Partial)** | SUM/COUNT on encrypted numeric fields using Paillier or BFV (research spike first). | `src/cryptodb/crypto/he.py` |
| 6.3 | **Geofencing** | Restrict decryption to specific geographic / network boundaries. | `src/cryptodb/auth/geo.py` |
| 6.4 | **Compliance Reports** | GDPR Right-to-erasure reports, HIPAA access reports, SOC2 evidence export. | `src/cryptodb/compliance/` |
| 6.5 | **Replication & HA** | Encrypted replication to standby nodes; ledger replication with consensus. | `src/cryptodb/replication/` |
| 6.6 | **Hardware Token Support** | YubiKey / TPM integration for KEK protection. | `src/cryptodb/crypto/hsm.py` |

## 6. Security Threat Model (Summary)

| Threat | Mitigation |
|--------|------------|
| Data at rest stolen | Envelope encryption + AEAD |
| Memory dump exposes keys | Keys in memory only during operation; secure zeroing |
| Insider abuse | Immutable ledger + RBAC + anomaly detection |
| Ledger tampering | Merkle hash chain + periodic signed checkpoints |
| Replay attacks | Nonces + timestamps in ledger |
| Privilege escalation | Principle of least privilege, ACLs |
| Backup exposure | Backups encrypted with distinct KEK |

---

## 7. Tests / Validation

- **Unit:** Every crypto primitive tested with NIST vectors and property-based tests.
- **Integration:** Full write -> read -> audit -> verify pipeline.
- **Fuzz:** Random payloads and concurrent access via `hypothesis` + `pytest-asyncio`.
- **Audit Verification:** Deliberately corrupt ledger file; ensure `verify.py` catches it.
- **Compliance:** Automated report generation checked against expected schema.

## 8. Risks, Tradeoffs, and Open Questions

| Risk | Mitigation |
|------|------------|
| Searchable encryption leaks partial info | Document tradeoff; offer opt-in per field |
| Key loss = data loss | KEK backup in HSM/shamir secret sharing |
| Performance of encrypted queries | Benchmark early; add caching layer if needed |
| Complexity of Phase 6 features | Gate behind feature flags; implement only after Phase 1вҖ“5 stable |

**Open Questions:**
1. Should the audit ledger be blockchain-anchored (e.g., Bitcoin OP_RETURN) for external notarization, or is local Merkle sufficient? USER: Local Merkle sufficient
2. Is multi-tenancy required from day one, or can it be added later? USER: It can be added later
3. What is the target deployment environment вҖ” local desktop, on-prem server, or cloud VM? USER: local desktop for testing, then on-prem server or cloud VM for deployment (it should be a portable python package)
